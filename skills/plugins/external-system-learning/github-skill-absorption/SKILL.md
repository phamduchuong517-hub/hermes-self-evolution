# GitHub 技能搜索与吸收进化方法论

从 GitHub 搜索高价值开源项目，提取核心能力，创建可执行脚本并更新本地技能的全流程。

---

## 先决条件

- `gh` CLI 已安装并认证（`gh auth status` 确认）
- 本地技能目录在 `~/.hermes/skills/`

---

## 阶段 1：搜索（找对项目）

### 搜索方法

| 方法 | 命令 | 适用场景 |
|------|------|---------|
| **gh api（推荐）** | `gh api "search/repositories?q=关键词&sort=stars&order=desc&per_page=5" --jq '.items[] \| "\(.stars)⭐ \| \(.full_name) \| \(.description)"'` | 最稳定，不走 curl |
| **避免用** | `curl -s "https://api.github.com/search/..."` | 云服务器上 curl 易超时（Clouflare 限流） |

### 关键词策略

每个技能方向搜 2-3 个不同的关键词，确保覆盖：

| 目标技能 | 搜索词 |
|---------|--------|
| self-improvement | `self-improving+agent`, `agent-evolution`, `self-modifying` |
| task-orchestrator | `task-orchestration+agent`, `workflow+orchestrator+ai` |
| skill-lifecycle | `skill+lifecycle+agent`, `skill-manager`, `agent-skillset` |
| token-optimization | `token-optimization`, `context-compression`, `prompt-compression` |
| evaluation | `agent-evaluation`, `skill-eval`, `llm-evaluation-framework` |

### 快速筛选标准

```
⭐⭐⭐⭐⭐ 直接吸收 — 相关度高 + 代码可用 + 成熟度好
⭐⭐⭐⭐  值得参考 — 有独特思路但需要适配
⭐⭐⭐    备选 — 偏学术/概念验证/生态依赖重
```

**注意**：不要只看 Star 数。skill-conductor 只有 89⭐ 但评估体系来自 Anthropic 官方改进版，比很多高 Star 项目质量更高。检查 README 中的"来源/借鉴"声明来判断真实质量。

## 阶段 1.5：必要性过滤（新增 — 推荐安装前必须先过此关）

**在推荐任何安装之前，先做必要性评估。** 不通过此关的项目绝不建议安装。

### 必要性过滤器

回答以下问题，**全部为"否"才能推荐安装**：

| 问题 | 回答 |
|:-----|:----:|
| 用户有实际使用这个能力的场景吗？（不是"将来可能有用"） | 是/否 |
| 该能力能否解决用户当前的一个明确痛点？ | 是/否 |
| 用户现有的工具/技能是否已经覆盖了这个能力？ | ⚠️ **有则跳过** |
| 项目的安装/运行环境与用户实际环境匹配吗？（Linux? 阿里云网络限制？） | 是/否 |
| 是否需要额外的 API Key/付费服务？ | ⚠️ **需要则谨慎评估** |

**如果"必要"清单的答案有任何一项是"否"，结论就是❌不需要安装——这是投机性安装，违反 Karpathy 原则。**

### 怎么做必要性分析

1. **回溯用户最近 5-10 次会话**——用户的真实工作流是什么？
2. **用户的核心任务分类**：编码、运维、写作、还是其他？
3. **对比用户的任务需求和能力缺口**——缺口不是由 GPT 想象出的，而是从历史记录中归纳的
4. **结果输出格式**：
   ```
   | 项目 | 必要性 | 理由 |
   |:---|:---:|:-----|
   | project-A | ✅ 需要 | ... |
   | project-B | ❌ 不需要 | 用户没有数据库项目 |
   ```

### 常见违规模式

| 违规模式 | 表现 | 修正 |
|---------|------|------|
| **投机安装** | "这个工具很好，虽然你现在用不上" | 用户用不上就别推荐 |
| **填补想象缺口** | "你在数据库方面是空白的，需要填空" | 空白 ≠ 需要，用户不需要的空白就是浪费 |
| **功能膨胀** | "装了这个还能做 X/Y/Z" | 只要核心场景不需要，附加功能再多也没用 |
| **忽略环境约束** | "这个爬虫很强" → 但用户阿里云服务器被 TCP 屏蔽，爬了也拿不到 | 环境约束是硬限制 |

### 必要性分析的最终输出

```
## 🎯 结论：[不需要/需要吸收任何项目]

| 检查项 | 结果 |
|:---|:---:|
| 有核心需求缺口吗？ | ✅/❌ |
| 现有技能覆盖不了？ | ✅/❌ |
| 装了能立刻用？ | ✅/❌ |
| 性价比高？ | ✅/❌ |
| **综合结论** | 🟢 保持现状 / 🟡 可考虑 / 🔴 优先吸收 |
```

## 阶段 2：分析（提取可吸收能力）

### 读取顺序

1. **README.md** — 了解项目是什么、解决什么问题、架构概览
2. **关键文件** — 看 `src/` 或 `scripts/` 下的核心逻辑，关注与本地相关的模块
3. **架构决策** — 项目的核心思路 vs 只是实现细节

### 对比框架

对比本地版本 vs 外部项目时，回答 5 个问题：

| 问题 | 判断 |
|------|------|
| 本地是否已有类似能力？ | ✅ 已有且更成熟→不吸收 |
| 外部项目提供了什么新能力？ | 新能力是否有实际使用场景？ |
| 新能力是否可代码化？ | 能做独立脚本才算真正吸收 |
| 需要多少适配工作？ | 从外部 API/依赖中解耦 |
| 吸收后能立即用吗？ | 要有执行入口（CLI 命令） |

### 吸收决策矩阵

| 场景 | 行动 |
|------|------|
| 外部有完整可执行代码 + 本地无 | **高价值代码吸收**：创建脚本 |
| 外部有完整系统架构 + 本地可参考 | **架构模式吸收**：提取设计模式 + P0/P1/P2 优先级矩阵，扩展现有技能的架构部分 |
| 外部有方法论/流程 + 本地无对应 | **方法论吸收**：创建/更新 SKILL.md（方法即产物，不一定需要代码） |
| 外部有方法 + 本地已有较弱版本 | **中价值吸收**：更新 SKILL.md 方法 |
| 外部只有理念 + 本地已有 | **低价值**：记个备注就行 |
| 外部依赖生态锁死 | **不吸收**：标记"值得参考但不兼容" |

### 判断被吸收对象的类型

在开始吸收前，先判断目标是"单一技能"还是"完整系统"：

| 特征 | 单一技能 | 完整系统 |
|------|---------|---------|
| 典型项目 | 一个库、一个 CLI 工具、一个功能包 | 桌面应用、框架、平台、Agent harness |
| 例子 | skill-conductor, headroom | tinyhumansai/openhuman, obra/superpowers |
| 吸收重点 | 可执行脚本、具体算法 | 设计模式、架构决策、分层结构 |
| 典型产出 | scripts/*.py 新脚本 | 扩展现有技能的架构部分、P0/P1/P2 矩阵 |
| 验收标准 | `python3 script.py` 能跑 | "我们的 SKILL.md 是否体现了更好的架构？" |

完整系统吸收的完整流程参见 parent skill `external-system-learning` 的"完整系统架构模式吸收"章节。

## 阶段 3：吸收（创建可执行脚本或更新技能文档）

### 核心原则

**代码吸收 = 可执行的真吸收，方法论吸收 = 可传承的真吸收。**

| 吸收形态 | 适用场景 | 验收标准 |
|---------|---------|---------|
| **代码脚本** | 功能可独立执行、有 CLI 入口 | `python3 script.py --help` 可用 |
| **架构模式** | 完整系统的设计思想 | SKILL.md 体现了更好的架构决策 |
| **方法论流程** | 流程/规范/步骤体系 | SKILL.md 给出了可复现的步骤 |
| **纯理念笔记** | 只是想法没有具体方案 | 不算吸收——记备注即可 |

⚠️ 注意：纯更新 SKILL.md 的文字描述而不产生产出物（代码/模板/配置文件）仍然不算真吸收。方法论吸收也必须产出**可复现的步骤序列**，不能只是"概念不错"的笔记。

### 脚本类型

| 类型 | 示例 | 适用场景 |
|------|------|---------|
| **评估脚本** | grader.py, comparator.py | 技能/输出质量评估 |
| **转换脚本** | compress.py, format.py | 数据转换/优化 |
| **分析脚本** | analyzer.py | 对比分析/根因分析 |
| **生成脚本** | init_skill.py | 模板/脚手架生成 |

### 脚本标准

每个脚本必须：
- ✅ 有 `argparse` 参数解析（CLI 可用）
- ✅ 有 `--verbose` 详细输出
- ✅ 有 `--output` 输出到文件
- ✅ 返回结构化 JSON（可被其他工具消费）
- ✅ 纯 Python stdlib，零外部依赖（除非真需要）
- ✅ 函数有 docstring 说明用途

### 目录结构

```
skills/<skill-name>/
├── SKILL.md              # 更新版本号 + 新增说明
├── scripts/
│   ├── grader.py         # 新脚本
│   └── comparator.py
├── evals/
│   └── evals.json        # 测试用例定义
```

## 阶段 4：验证（确认能用）

### 验证链

```bash
# 1. 语法检查
python3 -c "import ast; ast.parse(open('scripts/grader.py').read()); print('✅ 语法正确')"

# 2. 运行测试（用自身作为输入）
python3 scripts/grader.py evals/evals.json SKILL.md scripts/ -v

# 3. 对比测试
python3 scripts/comparator.py file_a.md file_b.md "对比任务"

# 4. 分析测试
python3 scripts/analyzer.py comparison.json skill_a.md skill_b.md
```

### 迭代调试循环（多特性 CLI 工具专属 ⭐）

构建像 `knowledge-memory.py` 这类多子命令的 CLI 工具时，不可能一次写完所有功能。应采用**循环迭代模式**：

```bash
循环体：
1. 实现一个子命令（功能完整但可优化）
2. 实际运行 python3 script.py subcmd --flags
3. 检查输出是否符合预期
4. 有 bug → patch 修复 → 回到 2
5. 功能正常 → 实现下一个子命令 → 回到 1

关键原则：
- 每个子命令实现后立即实测，不要堆砌完所有功能再测试
- 优先修"核心管道"（写入→持久化→读取）的端到端路径
- 验收顺序：基础功能（stats）→ 写入（ingest）→ 读取（query）→ 聚合（build-topic）→ 输出（global-digest）
- 边角优化（实体正则精度、格式化输出）留在核心管道跑通后
```

**2026-05-22 教训**: knowledge-memory.py 构建过程中 iterated through: chunker → SQLite → CLI → entity extractor → topic tree → global digest. 核心管道在第一次 ingest+stats 后就通了，但 entity extractor 的正则精度、关系查询的 SQL 性能、build-topic 的关联逻辑各修了 2-3 次。如果一次性写完全部功能再测试，debug 会非常痛苦。

### e2e 验收流程（自动清理测试数据）

```bash
# 多子命令 CLI 工具的端到端测试
rm -f ~/.hermes/memory/memory_knowledge.db  # 清理旧数据

# 1. 空数据库启动
python3 script.py stats

# 2. 摄入一条内容
python3 script.py ingest chat \
  --source_key "test-001" --title "测试" <<< "测试内容"

# 3. 验证写入
python3 script.py stats                    # chunks=1, entities>0
python3 script.py query "entity_name"      # 有返回值
python3 script.py source chat              # 能看到内容

# 4. 验证聚合
python3 script.py build-topic "topic_name" --force
python3 script.py topic "topic_name"       # 3 层摘要

# 5. 验证输出
python3 script.py global-digest --days 7   # 非空输出

# 6. 清理
rm -f ~/.hermes/memory/memory_knowledge.db
```

### 通过标准

- 所有脚本无 `SyntaxError` / `ImportError` 运行成功
- 输出有效 JSON（`json.loads()` 可解析）
- 评估结果合乎逻辑（不出现荒谬的判断）
- SKILL.md 更新了版本号和添加日期

## 经验教训

### 0. 方法论吸收 ≠ 代码吸收（新增）
**"文档吸收 = 僵尸吸收"太绝对了。** 有些技能本身就是方法论（如 brainstorming、mcp-builder）——核心价值是流程、原则、设计思路，不是算法代码。这类吸收就是产出 SKILL.md，不需要脚本。

**判断标准：** 这个能力的价值是在"怎么想"还是"怎么做"？
- 怎么想 → 方法论吸收（SKILL.md 就够了）
- 怎么做 → 代码吸收（需要脚本）

两者都叫真吸收，只是形态不同。

### 0.5 深度分析流（新增）
吸收 obra/superpowers (195K⭐) 这种巨型项目时，不要让吸收面发散。按以下步骤：

**Step A: 架构概览** — README + 目录结构
**Step B: 技能全景** — 列出对方所有技能，对比本地已有技能
**Step C: 吸收决策矩阵** —
| 对比结果 | 行动 |
|----------|------|
| 对方有 + 我们无 | 列为吸收候选 |
| 对方有 + 我们有但弱 | 融合升级本地技能 |
| 对方有 + 我们已超越 | 跳过 |
| 对方有 + 不适用场景 | 说明原因，明确跳过 |
**Step D: 按优先级执行** — P0 → P1 → P2
**Step E: 记录吸收来源** — 更新 ABSORPTION-RECORD.md

### 1. 搜索用 gh api 不要用 curl
`gh api` 走的是 GitHub 认证通道，比直接 `curl GitHub API` 稳定得多。这台云服务器上 curl 访问 api.github.com 频繁超时。

### 2. 先验证代码吸收可行性，再更新 SKILL.md
我这次先写代码再更新文档的顺序是对的。如果先花时间更新了 SKILL.md 再发现脚本跑不通，得回头改两次文档。

### 3. 不要为所有新能力都去写脚本
有些能力（如 HyperAgents 的"自引用改进回路"）是学术概念性，不适合代码化吸收。只对"可执行"的能力创建脚本。

### 4. evals.json 是很好的"需求文档"
在写脚本之前先定义 evals.json，相当于先定义验收标准再写实现——天然 TDD。

### 5. 记下每次吸收的"来源"和"日期"
每个被吸收的 SKILL.md 末尾加吸收来源表格，方便追溯哪些能力来自哪些项目。

## 阶段 1.7：吸收成果质量评估（新增 ⭐）

**适用场景**: 有人（如其他子Agent、外部系统）已经完成了某个项目的吸收并产出了技能，你需要**评估这个吸收成果的质量**，而不是自己做吸收。

### 评估框架

| 维度 | 满分 | 问题 | 检查方法 |
|------|:----:|------|---------|
| **完整性** | 10 | 是否覆盖了目标项目的核心能力？是否遗漏了关键项目？ | 对照GitHub搜索，看该技术方向有几个高Star项目，被评估的技能覆盖了几个 |
| **准确性** | 10 | 技术原理描述是否和文档一致？架构图是否正确？ | 读目标项目的README对比，检查核心设计是否写对 |
| **实用性** | 10 | 是否回答了"这项目在我的场景下能直接用吗"？是否有部署建议？ | 看有没有安装必要性分析、环境要求检查、与现有工具的对比 |
| **可执行性** | 10 | 看完技能我知道下一步该做什么吗？还是只是堆砌了信息？ | 看结论是否明确（装/不装/什么时候装），有无行动指引 |
| **深度** | 10 | 是否理解了设计的"为什么"而非只是"是什么"？ | 检查有没有分析技术权衡（为什么选这个方案？替代方案是什么？） |

**总分解释**:
| 分数 | 评价 |
|:----:|:-----|
| 40-50 | 🟢 高质量吸收，可以直接参考 |
| 30-39 | 🟡 基本可用，有1-2个明显缺失需要补 |
| 20-29 | 🟠 偏离了重点，关键部分缺失，建议重写 |
| <20 | 🔴 信息堆砌，没有实际价值 |

### 常见缺陷模式

| 缺陷 | 表现 | 修正方向 |
|------|------|---------|
| **只堆不判** | 把所有项目的信息堆在一起，但不说"所以呢？" | 加结论行 + 可行性表 |
| **缺必要分析** | 没有检查环境匹配度、用户场景、安装前提 | 加必要性评估（阶段1.5） |
| **偏漏** | 高Star实用项目没列，低Star学术项目却深度分析 | 统一按Star数+实用性双维排序 |
| **Vendor Lock 幻觉** | 把项目说得天花乱坠，没提需要服务器/API Key/付费 | 加环境约束检查 |
| **技术膨胀** | 列出了4个组合策略（如策略A/B/C/D），但用户实际上一个也用不上 | 每个策略标清楚先决条件 |

### 输出格式

```markdown
## 吸收成果质量评估

| 项目 | 核心能力 | 评估 |
|:-----|:---------|:----:|
| project-A | 核心原理 | ✅ 正确 |
| project-B | 核心原理 | ⚠️ 缺了X细节 |
| ... | ... | ... |

**完整性**: X/10 — 覆盖了Y个方向中的Z个，遗漏了...
**准确性**: X/10 — 描述是否与README一致
**实用性**: X/10 — 能否直接用于当前场景
**可执行性**: X/10 — 看完知道该不该装
**深度**: X/10 — 理解了设计权衡吗

**总分**: X/50 — 🟢/🟡/🟠/🔴

### 主要缺失
1. 缺失项A — 建议补上
2. 缺失项B — 如果重要

### 最终结论
[一句话总结：可用/需要补丁/建议重写]
```

### 实例（来自2026-05-29 `vpn-core-tech` 技能评估）

# 🧠 Brainstorming — 设计细化工作流

**位置**: grill-with-docs（需求对齐）→ **brainstorming（设计细化）** → writing-plans（实施计划）

## 为什么需要这个技能

grill-with-docs 回答了"要做什么"，brainstorming 回答"具体怎么做"。

我们有一个问题：经常在需求明确后直接跳入 writing-plans（甚至直接写代码），跳过了**设计探索**阶段。这导致：
- 编码到一半发现架构不对要重来
- 没有设计文档，未来的人看不懂为什么这么设计
- 假设悄悄变成"事实"，最后发现理解不一致

## 触发条件

在以下情况**必须**触发（在 grill-with-docs 之后、writing-plans 之前）：

- 开始一个新的功能/组件/项目——无论自认为多简单
- 修改不熟悉代码库中的核心模块
- 需求明确但实现路径不确定

> **"太简单了不需要设计"是反模式。** Todo list、单函数工具、配置变更——所有项目都过一遍设计。所谓"简单"的项目恰恰是未检视假设导致最多返工的地方。

## 核心流程

### 1. 探索项目上下文

检查当前项目状态：文件结构、现有模式、近期改动。

### 2. 澄清问题

一次只问一个问题，逐条推进：

- 目标是什么？成功的验收标准？
- 关键约束？（性能/安全/兼容性）
- 边界在哪里？哪些明确不做？
- 有什么已有的模式需要遵循？

**一次只问一个问题**。不要同时扔多个问题给用户。

### 3. 提出 2-3 种方案

给出 2-3 种不同方案及其权衡，附上你的推荐。

**格式**:
```
## 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| A（推荐） | ... | ... |
| B | ... | ... |
```

先说出你的推荐，然后解释为什么。

### 4. 展示设计

分段展示设计，每段根据复杂度调整长度（简单→几句话，复杂→200-300字）。

每展示一段，问用户"这部分对吗？"——逐步验证。

覆盖：
- 架构 / 组件划分
- 数据流
- 错误处理
- 测试策略
- 与现有代码的集成点

### 5. 隔离设计原则

**每个单元只做一件事**，通过清晰的接口通信，能被独立理解和测试。

```
对一个好的单元，你能回答：
- 它是做什么的？
- 怎么用？
- 依赖什么？
```

**文件膨胀通常是做得太多的信号。**

### 6. 保存设计文档

用户确认设计后，保存到 `docs/superpowers/specs/YYYY-MM-DD-主题-设计.md`。

如果用户有自定义路径偏好，按偏好来。

### 7. 自审设计文档

写完文档后，用新鲜眼光审视：

1. **占位符扫描**：有无 "TBD"、"TODO"、不完整章节？
2. **内部一致性**：各章节是否矛盾？架构是否匹配功能描述？
3. **范围检查**：是否聚焦到单个计划的程度，还是需要进一步分解？
4. **歧义检查**：任何需求是否可能被两种不同方式理解？

发现即修，无需再审查。

### 8. 用户审阅设计文档

```
设计文档已写入 `<路径>`。请审阅并告诉我是否需要修改，然后我们开始制定实施计划。
```

等待用户回应。如需修改则修改后重新自审。用户批准后才进入 writing-plans。

## 流程图

```
探索项目上下文 → 逐条追问 → 提出 2-3 方案
     → 展示设计（逐段确认）
     → 用户批准？
       ├─ 否 → 回到展示设计
       └─ 是 → 保存设计文档
              → 自审（修正内联）
              → 用户审阅文档
                ├─ 需改 → 修改并重新自审
                └─ 批准 → 进入 writing-plans
```

**终点状态是调用 writing-plans。** 不调用其他实现技能。

## 设计文档模板

```markdown
# [功能名称] — 设计文档

**日期**: YYYY-MM-DD
**状态**: 已批准

## 目标
一句话说明这个功能要解决什么问题。

## 架构
2-3 句描述整体方案。

## 组件
- 组件 A: 职责 + 接口
- 组件 B: 职责 + 接口

## 数据流
描述数据如何流经各组件。

## 错误处理
关键错误场景及处理方式。

## 测试策略
哪些需要测试、怎么测。

## 非目标
明确不做的事。
```

## 关键原则

- **一次一个问题** — 不要连珠炮
- **方案对比** — 总是给 2-3 种选择
- **YAGNI** — 从设计中砍掉所有非必要功能
- **逐步确认** — 每段设计都问"这部分对吗"
- **文档存盘** — 批准的设计必须写成文档提交

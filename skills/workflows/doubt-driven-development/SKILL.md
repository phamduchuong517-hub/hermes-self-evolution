# 🧐 Doubt-Driven Development — 零上下文审阅

**吸收自**: addyosmani/agent-skills (42k⭐, 2026-05-17)
**来源**: Addy Osmani (Google Chrome 工程总监)

## 为什么需要这个技能

> 一个**自信的答案**不等于一个**正确的答案**。

长会话会积累上下文，悄悄把假设变成"事实"而没人注意到。这个技能的纪律是：在任何重要输出成型之前，请一个**零上下文的审阅者**——它的偏见是**证伪**，不是**批准**。

这不是 `/review`（对已完成的产物的评估），这是一个**飞行中的姿态**：在还能低成本改方向的时候交叉验证。

## 触发条件

**非重要决策**至少满足以下一条：

- 引入或修改了分支逻辑
- 跨模块/服务边界
- 断言了类型系统/编译器无法验证的属性（线程安全、幂等性、排序、不变量）
- 正确性依赖于未来读者看不到的上下文
- 影响面不可逆（生产部署、数据迁移、公开 API 变更）

**应用时机**：
- 即将在不确定下做架构决策
- 即将提交非平凡的代码
- 即将声称一个非显而易见的"事实"（"这个安全""这个能扩展""这个匹配需求"）
- 在不完全理解的代码中工作

**不要应用**：
- 机械操作（重命名、格式化、移动文件）
- 遵循清晰明确的操作指令
- 阅读或总结现有代码
- 一行变更且正确性显而易见
- 纯工具操作（跑测试、列文件）
- 用户明确要求速度胜过验证

## 核心流程：5 阶段审阅

### Step 1: CLAIM — 显式化你的断言

把决策写成 2-3 行：

```markdown
CLAIM: "新的缓存层在 spec 描述的读密集工作负载下是线程安全的"
WHY THIS MATTERS: 竞态会损坏用户数据，且在 QA 中难以检测
```

如果你写不出这么紧凑的声明——说明这是个直觉，不是决策。说出来再审视它。

### Step 2: EXTRACT — 最小可审阅单元

零上下文审阅者需要的不是"思考过程"，而是：
- **产物**: diff 或函数本身（不是整个文件）
- **契约**: 3-5 句话的提议 + 它必须满足的约束

```markdown
## ARTIFACT
{代码 diff 或关键函数}

## CONTRACT
{提议描述：做什么、怎么做、约束条件}
```

关键：**去掉推理过程**。审阅者需要知道"你做了什么"，不是"你经历了什么思考"。

### Step 3: DOUBT — 零上下文审阅

**这是核心机制**。关键元素：

1. **零上下文** — 审阅者是全新的 Agent 进程，没有任何之前的会话上下文
2. **偏见设置为证伪** — 审阅者的目标是找到一个理由说"这是错的"
3. **只给产物+契约** — 不给推理过程，不给历史

如果子 Agent 不可用（技术限制），降级为一个**硬心理分隔**的自我审阅——在同一个会话中假装自己是第一次看到这个代码。标记为降级。

**审阅指令模板**：

```markdown
你是一个代码审阅者，你的任务是**证伪**下面的方案。你的偏见是"这是错的"——你的目标是找到错误。

你没有任何上下文，你只知道下面这些：

## 产物
{代码/决策}

## 契约
{做什么、怎么做、约束}

你的任务：
1. 找出所有可能出错的地方
2. 假设每个假设都是错的，找反例
3. 指出逻辑漏洞、边界情况、隐藏依赖

记住：你的工作是找到问题，不是让人感觉良好。
```

### Step 4: RECONCILE — 调合

审阅者返回结果后：

1. 逐条分类每条发现：
   - **真实问题** → 修复
   - **误解** → 澄清（审阅者缺少上下文导致的误判）
   - **不相关** → 忽略
2. 修复真实问题，更新产物
3. 输出调合结果

```markdown
## 调合

| 发现 | 分类 | 行动 |
|------|------|------|
| 未处理 null 输入 | 真实问题 | 已添加 null check |
| X 应该用 Y 替代 | 误解（Y 已弃用） | 已澄清 |
| 性能可能有问题 | 需要基准测试 | 新建 issue 跟踪 |
```

### Step 5: STOP — 停止条件

满足任一即停止：

1. **审阅发现全是误解** — 所有问题都是审阅者缺少上下文导致的，没有真正的问题
2. **最多 3 轮** — 每轮审阅+修复算一轮，3 轮后无论怎样都停
3. **用户覆盖** — 用户说"别审了，就这样"

## 使用示例

### 场景：缓存架构决策

```
用户: "给用户服务加一个 Redis 缓存层"

Agent:
Step 1 CLAIM:
  "基于 Redis 的缓存层，缓存用户资料查询，TTL 5 分钟，缓存穿透时直接查 DB"

Step 2 EXTRACT:
  产物: 缓存拦截器代码
  契约: 读密集工作负载下减少 DB 压力

Step 3 DOUBT:
  零上下文审阅者发现:
  - ❌ 缓存穿透时直接查 DB → 没有做互斥，高并发下会打垮 DB
  - ❌ TTL 5 分钟 → 某些场景（封号）需要即时失效，没考虑

Step 4 RECONCILE:
  - 加锁穿透 → 修复
  - 主动失效机制 → 修复

Step 5 STOP:
  → 2 轮后无新发现 → 停止
```

## 与 /review 的区别

| | **Doubt-Driven** | **Code Review** |
|:-|:-|:-|
| 时机 | 飞行中（代码还没定型） | 完成后（PR 阶段） |
| 审阅者上下文 | 零上下文（全新进程） | 完整上下文 |
| 偏见 | 证伪（找错误） | 平衡（找问题+确认正确） |
| 产物 | 最小单元（函数/决策） | 完整 PR diff |
| 成本 | 低（小范围快速） | 高（大范围深入） |

**互补使用**：编码中用 DDD 快速验证关键决策，PR 时用 code review 做完整审计。

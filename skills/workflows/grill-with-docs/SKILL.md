# 🎯 Grill With Docs — 深度追问与术语对齐

**吸收自**: mattpocock/skills /grill-with-docs (87k⭐, 2026-05-17)
**来源**: Matt Pocock (Total TypeScript 作者，TypeScript 社区领袖)

## 为什么需要这个技能

Agent 开发最常见的失败模式是**对齐失败**。你以为你说清楚了，Agent 以为它理解了——结果做出来的东西和你想的不一样。

这个技能通过在编码前进行一次**系统化的深度追问**来消除这个差距，同时建立项目的共享语言（CONTEXT.md），让之后的每次对话都更精确、更省 token。

## 触发条件

在以下场景自动触发：
- 开始一个**新功能/新项目**的编码前
- 用户说"我想做一个..."但需求模糊
- 需要在**不熟悉的代码库**中做修改
- 发现 Agent 和你的理解有偏差

## 核心流程

### Phase 1: 追问（Grilling）

逐条追问，**一次只问一个问题**，等待回答后继续。

追问清单（按优先级排列）：

**1. 目标澄清**
- "你想要达到什么具体效果？"
- "成功的标准是什么？怎么验证做完了？"

**2. 约束条件**
- "有什么限制？（性能/安全/兼容性/预算）"
- "有没有必须兼容的技术栈版本？"

**3. 边界定义**
- "这个改动会影响哪些模块？"
- "哪些是明确不需要做的？"

**4. 风险识别**
- "你最担心什么地方出错？"
- "如果这个搞砸了，最坏情况是什么？"

**5. 决策树**
- "关于 X，你觉得是方案 A 还是方案 B？"
- "有什么你已经有倾向但不确信的决定？"

**每问完一个问题，给出你的推荐答案**，让用户确认或反驳。

### Phase 2: 术语对齐

在追问过程中，**时刻留意术语冲突**：

- 当用户用了一个和现有代码/文档冲突的词 → 立即指出
- 当用户用了模糊的通用词（"账号""用户""项目"）→ 要求精确
- 当某个概念用多个词混用 → 确定标准术语

**输出**: 更新或创建 `CONTEXT.md`（项目共享语言文档）

#### CONTEXT.md 格式

```markdown
# {上下文名称}

{一两句话说明这个上下文是什么}

## 语言

**{标准术语}**:
{一句话精确定义}
_避免_: {同义词列表，用这些就错了}

**{另一个术语}**:
{定义}
_避免_: ...

## 关系

- **订单** 产生一个或多个 **发票**
- **发票** 属于且只属于一个 **客户**

## 标记的歧义

- "账号"被混用为 **客户** 和 **用户** → 已解决：这两个是不同的概念
```

**关键原则**：
- 共识优先于正确性（只要团队同意就是对的）
- 一条定义不超过一句话
- 只记录项目特有的概念，不记录通用编程概念
- 当自然聚类出现时分组（不是必须）

#### 多上下文

- 单一上下文：根目录 `CONTEXT.md`
- 多上下文：`CONTEXT-MAP.md` 列出所有上下文及其关系
- 工具会探测：有 `CONTEXT-MAP.md` 就读它，否则看根 `CONTEXT.md`，都没有就首次对齐时创建

### Phase 3: ADR 决策记录

**只在以下三个条件同时满足时才创建 ADR**：

1. **难以逆转** — 改主意成本高
2. **不看上下文会惊讶** — 未来的人看代码会问"为什么这么搞？"
3. **经过了真正的权衡** — 确实有备选方案且你选了某个

**ADR 格式**（简洁主义，不写废话）：

```markdown
# {简短标题}

{1-3 句话：上下文是什么，决定了什么，为什么}
```

**可选段落**（只在有必要时加）：
- **Status**: proposed | accepted | deprecated | superseded by ADR-NNNN
- **Considered Options**: 被拒绝的备选值得记录时
- **Consequences**: 有非显而易见的连锁影响时

**什么值得写 ADR**：
- 架构形状（"我们用 monorepo"）
- 上下文间的集成模式（"订单和账单通过领域事件通信，不是同步 HTTP"）
- 带有锁定成本的技术选型（数据库、消息队列、认证提供方）
- 边界和范围决策（"客户数据由客户上下文持有，其他上下文只引用 ID"）
- 故意偏离常规的做法（"我们用手写 SQL 而不是 ORM，因为 X"）
- 代码中看不到的约束（"不能用 AWS 因为合规要求"）

ADR 存放在 `docs/adr/`，编号格式 `0001-标题.md`，按顺序递增。目录在首次需要 ADR 时懒创建。

## Phase 4: 总结

追问+对齐完成后，输出：

```markdown
## ✅ 对齐摘要

**目标**: {一句话}
**关键术语**: {3-5 个核心术语及其定义}
**ADR**: {本次创建的 ADR 列表，无则无}
**下一步**: {行动项}
```

## 使用示例

### 场景：开始一个新功能

```
用户: "帮我加一个用户积分系统"

Agent:
1. 追问目标 → "积分怎么获取？消费1元=1分？"
2. 术语对齐 → "你说的'积分'和代码里已有的'点数'是一个概念吗？"
3. 边界定义 → "积分有没有有效期？过期逻辑怎么做？"
4. 更新 CONTEXT.md → 新增"积分""兑换""等级"等术语
5. ADR → 如果选了特定的持久化方案，记录 ADR
6. 输出对齐摘要
```

### 场景：修改不熟悉的代码库

```
用户: "改一下这个支付流程"

Agent:
1. 读 CONTEXT.md(如果有) → 了解共享语言
2. 追问 → "你说的改支付流程是指改网关对接、还是改UI、还是改后端逻辑？"
3. 读 ADR → 了解为什么当初这么设计
4. 术语对齐 → "代码里同时出现了 Payment 和 Transaction，是一个东西吗？"
```
